\chapter{Evaluation}\label{ch:evaluation}

In this chapter we will briefly reintroduce the language design concepts we discussed earlier and see how they fit into simple examples of real-world contracts. These example contracts stem from a widely-used smart contract library maintained by OpenZeppelin. They often only implement a very specific functionality and therefore usually remain quite small and easy to understand. We have reimplemented the identical functionality in our language to compare and discuss the differences. \\
Afterwards, we will look at some vulnerabilities that were exploited in well-known contracts. After seeing the same functionality reimplemented in our language, we will demonstrate the difficulty of writing code that is vulnerable in the same fashion. Of course, we cannot address all types of vulnerabilities, merely the ones we set out to solve initially.

\section{Algebraic Data Types}
When modelling a state machine, it is standard practice to make use of regular enumeration types. However, such a value can only encode a single state, but not contain any auxiliary information. Standard workarounds include using "magic values", i.e., encoding state information via setting a field to a specific predetermined value, as well as defining a "kitchen-sink" struct, namely one that contains all fields for all states at the same time.

For example, ERC20 tokens are specified such that the value \rustinline{u256::MAX} $= 2^{256}-1$ represents an unlimited allowance, whereas any other value represents an allowance equivalent to its actual quantity. Now, any read or write to such an allowance must be checked against this magic value, leading to a higher risk of developer mistakes. Here, we compare the "standard" implementation of the ERC20 \rustinline{transferFrom} function, as specified by the ERC20 token standard:

\begin{changemargin}{-0.7cm}{-0.7cm} 
\noindent\begin{minipage}[t]{.5\textwidth + 0.7cm}
\begin{lstlisting}[caption=Solidity, language=Solidity, style=boxedSol, basicstyle=\ttfamily\scriptsize]
contract ERC20 {
	mapping(address => uint256) private _balances;
	mapping(address =>
		mapping(address =>
			uint256)) private _allowances;

	function transferFrom(
		address from,
		address to,
		uint256 value
	) external returns (bool) {
		require(from != address(0));
		require(to != address(0));

		address spender = msg.sender;
		uint256 amount = _allowances[from][spender];
		if (amount < type(uint256).max) {
			_allowances[from][spender] = amount - value;
		}

		execute_transfer(from, to, value);
		return true;
	}
}
\end{lstlisting}
\end{minipage}
\begin{minipage}[t]{.5\textwidth + 0.7cm}
\begin{lstlisting}[caption=Ours, language=Rust, style=boxed, basicstyle=\ttfamily\scriptsize]
contract ERC20 {
	balances: HashMap<Address, u256>,
	allowances: HashMap<Address,
		HashMap<Address, Allowance>>,
}
impl ERC20 {
	pub fn transfer_from(self,
		from: Address,
		to: Address,
		value: u256
	) -> bool {
		assert(from != Address::ZERO);
		assert(to != Address::ZERO);

		let spender = EVM::caller();
		self.allowances[from][spender].spend(value);
		self.execute_transfer(from, to, value);
		return true;
	}
}
enum Allowance {
	Amount(u256),
	Unlimited,
}
impl Allowance {
	fn spend(self, value: u256) {
		match self {
			Allowance::Amount(amount) => {
				amount -= value;
			}
			Allowance::Unlimited => (),
		}
	}
}
\end{lstlisting}
\end{minipage}
\footnotesize
\emph{Note: In both cases, no underflow check is necessary, because both compilers insert overflow checks on arithmetic operations by default. However, it may sometimes be preferable to check manually in order to produce a more specific error message.}
\end{changemargin}

Defining an ADT in such a manner provides two distinct advantages: First, it becomes much simpler to separate the underlying logic (reducing the allowance by the spent amount) from the contract's functionality (transferring tokens). Second, the type system now makes it impossible to misinterpret an unlimited allowance as a finite one, whereas in Solidity the developer needs to actively remember to check for the "magic value" every time they use an allowance. \\
Of course, in such a simple example the benefits are dampened by the additional overhead induced by the additional code needed to define of the ADT. However, in more complex cases the advantage becomes much clearer. Furthermore, smart contracts often model processes that can be easily represented by a state machine. This is the perfect application for an ADT, as we will see in the next example.

\subsection{Real-world example}
The widely-used OpenZeppelin contract library features a \solinline{Governor} contract, which models voting processes. Users can submit proposals, each one of which can be in one of eight different states. Depending on the state, a proposal needs to keep track of different values. For instance, while voting is active, it needs to store the deadline for the end of the voting period, as well as the actual vote counts. This data is stored in the \solinline{ProposalCore} struct. Precisely which \solinline{ProposalState} the proposal is currently in is dictated by the values stored in the fields. The logic of determining this state is implemented by the \solinline{state(...)} function.
 
\begin{lstlisting}[language=Solidity, style=boxedSol]
\end{lstlisting}

\noindent\begin{minipage}[t]{.3\textwidth}
\begin{lstlisting}[caption=Proposal State Data, language=Solidity, style=boxedSol, basicstyle=\ttfamily\small]
struct ProposalCore {
	uint48 voteStart;
	uint32 voteDuration;
	bool executed;
	bool canceled;
	uint48 etaSeconds;
}
enum ProposalState {
	Pending,
	Active,
	Canceled,
	Defeated,
	Succeeded,
	Queued,
	Expired,
	Executed
}
\end{lstlisting}
\end{minipage}
\begin{minipage}[t]{.7\textwidth}
\begin{lstlisting}[caption=Proposal State Logic, language=Solidity, style=boxedSol, basicstyle=\ttfamily\small]
function state(uint256 id) returns (ProposalState) {
	if (proposals[id].executed) {
		return ProposalState.Executed;
	}
	if (proposals[id].canceled) {
		return ProposalState.Canceled;
	}
	
	uint256 start = proposals[id].voteStart;
	uint256 now = clock();
	if (start >= now) {
		return ProposalState.Pending;
	}
	
	uint256 deadline = start + proposals[id].voteDuration;
	if (deadline >= now) {
		return ProposalState.Active;
	} else if (!quorumReached(id) || !voteSucceeded(id)) {
		return ProposalState.Defeated;
	} else if (proposals[id].etaSeconds == 0) {
		return ProposalState.Succeeded;
	} else {
		return ProposalState.Queued;
	}
}
\end{lstlisting}
\end{minipage}

Recall that this is a widely-used \textit{library contract}, not just some one-off project! As you can see, even just determining what state the proposal is in becomes exceedingly complex. Instead, it would be much simpler to describe a proposal starting from its states, then only include the fields relevant to each state. This is precisely what can be done with an ADT. As an additional benefit, the compiler rejects reading field values that are not relevant to the current state. For some additional clarity, here is what such an ADT might look like, in Rust syntax. It also demonstrates well that most fields are irrelevant for most states.
\begin{lstlisting}[language=Rust, style=boxed]
enum ProposalCore {
    Pending {
        voteStart: u64,
    },
    Active {
        voteStart: u64,
        voteDuration: u64,
    },
    Canceled,
    Defeated,
    Succeeded,
    Queued {
        etaSeconds: u64,
    },
    Expired,
    Executed,
}
\end{lstlisting}

\section{Reentrancy Protection}

Reentrancy vulnerabilities are one of the most widely-known classes of smart contract bugs. Despite the fact that most smart contract developers are aware of their existence, reentrancies are to this day one of the most common vectors for real-world exploits. Thus, it is clear that the tools provided by the current development ecosystem are insufficient to systematically prevent this type of vulnerability.

In fact, Solidity provides no built-in reentrancy protection tools at all, although its \textit{function modifier} feature provides a convenient way for developers to implement their own protection mechanism. Vyper on the other hand does provide opt-in reentrancy protection, although even this protection was exploited due to a compiler bug.\\
Note that in both cases, the reentrancy checks are specific to their entrypoints. This means that if an entrypoint is not protected, any subsequent external call made by the contract could potentially reenter any of the contract's entrypoints. If an entrypoint \textit{is} protected, any external call made is still able to reenter all unprotected entrypoints of the contract.

It is trivial for the compiler to infer what kind of check needs to be made based on the entrypoint, as well as only allowing specific external calls to reenter only their desired entrypoints.

Only a minority of smart contracts even require reentrant calls for their core functionality, and even fewer of these require anything more than a single reentrant call path. As such, it seems prudent for most cases to disable reentrant calls by default, and allow the developer to individually define each required reentrant call path individually. If the core functionality of a smart contract must include reentrancies, the developer must consider the safety of such calls very carefully. Hence, it is little additional overhead to require them to specify each call path individually. In fact, it requires every necessary call path to be explicitly considered, so it is virtually impossible for an unexpected reentrant call to happen.

\subsection{Real-world examples}

There is an abundance of real-world reentrancy vulnerabilities, although many are either outdated or part of large projects and hence difficult to summarize succinctly.

In October 2022, a new class of reentrancy vulnerability was first described -- named a read-only reentrancy. It was long the consensus that reentrancy bugs could only occur if the reentrant call itself modified the contract state. However, simply the fact that a read-only function might return an incorrect intermediate value can be enough to trigger a bug. Most cases of such bugs involve price oracles -- that is, contracts whose purpose is to estimate the current exchange price between two or more currencies. It is often possible to manipulate these oracles, although it is usually at great expense. In some cases, it is possible to manipulate the oracle price, then for the oracle to call back to the user. If a third-party contract now relies on the value returned by the oracle contract, it will read the manipulated price and be vulnerable to exploitation. Note that this would not be possible if the oracle contract didn't allow control flow to reenter the read-only function.

In fact, the standard approach to reentrancy protection in Solidity does not even allow protecting read-only functions. Instead, developers must implement two separate checks, one for regular calls and one for read-only calls. Meanwhile, Vyper has extended its opt-in mechanisms to also include read-only functions as well.

\section{Side Effects}

Note: "Side effect" is a very broad term that can be applied to many types of behaviour. In this case, we specifically discuss \textit{state modifications}, but the approach can be extended to almost any other kind of side effect, such as state reads, event emission, external calls, and so on.\\
In a sense, one could say that the purpose of smart contracts is to cause side effects on the blockchain. In any case, it is quite important for developers to be aware of what side effects are happening in what order. In the first case, hidden or implicit side effects in the code can trigger unexpected behaviour, for example if one state change overwrites another, or if outdated values are used for further computations. In the second case, ambiguities in execution order can also lead to unexpected behaviour, although this is more limited to \textit{intentionally misleading} code, such as the winning submission in the \textbf{Underhanded Solidity Contest} in 2022.

The evaluation order of expressions is explicitly stated to be undefined in Solidity, which means that a vast amount of deployed smart contracts are actually inducing undefined behaviour during the compilation process. Of course, this is almost always benign, but still represents a real threat. The Vyper documentation also explicitly states that the evaluation order of arguments to built-in functions is undefined. In other cases, the documentation does not specify anything at all.

Keeping track of state modifications throughout the various call paths of a smart contract is also critical in the case of smart contract audits. If it were at a glance clear that an entrypoint cannot trigger certain state modifications, it would be much easier to rule out certain kinds of vulnerabilities, without having to manually go through each line of code. Additionally, if the syntax is explicitly clear about when side effects can happen, it is much less likely for developers to overlook unexpected state modifications. For example, Vyper only allows state variables to be modified via syntax such as \lstinline{self.xyz = 5}. Contrast this to solidity, where a statement such as \lstinline{xyz = 5;} does not make immediately clear whether it is modifying a local variable or a state variable of the contract.

However, in both languages, the syntax of a function call does not explicitly distinguish whether or not state modifications can happen. When reading code, it is impossible to know what side effects might be caused by a function call without fully understanding the function's body. In a sense, this is equivalent to the concept of \textit{information hiding}, which means that a caller should not be reliant on the internals of a function, but instead only depend on its signature. Thus, we can require functions to explicitly state any possible modifications in their signatures.\\
Although this significantly improves the situation, it doesn't entirely solve the problem -- if we see a function call like \lstinline{self.foo()}, it's still necessary to go and look at the signature of the function to determine its declared side effects. Again, this may allow unintended side effects to go unnoticed. If the developer is also required to explicitly allow the side effects at the \textit{call site}, then it becomes immediately clear at a glance what exactly is happening.

\begin{lstlisting}[language=Rust, style=boxed]
pub fn foo(amount: u256) -> u256
  modifies(self.last_amount)
  modifies(self.n)
{
  self.last_amount = amount;

  let count: u256;
  modify(self.n) {
    count = self.increment();
  }

  return count;
}

fn increment() -> u256
  modifies(self.n)
{
  let result = self.n;
  self.n += 1;
  return result;
}
\end{lstlisting}

As you can see in the above example, only the indirect side effect caused by the call to \lstinline{self.increment()} is required to be explicitly declared. In the case of \lstinline{self.last_amoount = amount}, it is already clear that the state is being modified without any surrounding declarations.

\subsection{Side Effects in Reentrant Calls}

This concept of side effect declaration also pairs very well with the granular reentrancy path declaration detailed earlier. In fact, without such a feature, it would be almost impossible to determine what side effects might occur to the contract state during an external call. However, if the external call must specify exactly which reentrancies are possible, then the compiler can treat the external call as though it were a regular function call, at least concerning side effects. Thus, the call site must be marked with the potential side effects that would be induced if the reentrant call is made.\\
Note that we also require the possible reentrancy to be declared in the function signature. If the reentrant call itself may trigger further such calls, it should be clear at the call site that this is a possibility. However, it is unlikely for a real-world use case to strictly require nested reentrant calls, so it would also suffice to simply disallow further reentrant calls.

\noindent\begin{minipage}{.45\textwidth}
\begin{lstlisting}[caption=Ours, language=Rust, style=boxed]{Name}
pub fn foo(
  x: u256
) -> u256
  modifies(self.val)
  reenters(Self::bar)
{
  self.val = x;
  modify(self.val) {
    EVM::reentrant_call(
      Self::bar,
      address,
      data
    );
  }
  return self.val;
}

pub fn bar()
  modifies(self.val)
{
   self.val += 1;
}
\end{lstlisting}
\end{minipage}\hfill
\begin{minipage}{.45\textwidth}
\begin{lstlisting}[caption=Solidity, language=Solidity, style=boxedSol]{Name}
function foo(uint x)
  public
  returns (uint)
{
    value = newValue;
    address.call(data);
    return value;
}

function bar()
  public
{
    value += 1;
}
\end{lstlisting}
\end{minipage}

Here, it is again immediately obvious that the new value written to the state variable may be overwritten by the reentrant call. Contrast that to the equivalent Solidity code, where it is not even clear that a reentrant call is possible in the first place, let alone what side effects it may cause. In fact, any of the existing languages for EVM would be equivalent, since their reentrancy protections are declared at the entrypoints, not the call sites.\\
Of course, this example also makes clear that the strict requirements for side effect declarations result in verbose code. However, we believe that this is a trade-off which is worth the additional overhead, since the stakes are so high.
